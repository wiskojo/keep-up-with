# Output Format

Read this before writing a user-facing FYI, BTW, medium thread, or deep thread.

## Quick FYI / BTW

Use for lightweight updates.

- 1-4 sentences.
- Include the main link inline.
- State what happened and why the user is seeing it now.
- No headings, source dumps, or process narration.

Shape:

```text
{What happened.} {What changed or why it matters to this user.} {Uncertainty or next step if needed.} {Link}
```

## Medium / Deep Thread

Use a thread for researched updates. Keep the main channel clean; put sources in the thread.

```markdown
Background:
Who is involved? If it is a random startup, add context. If it is a known lab, company, repo, paper, or person, keep this short.
When did this happen? If it is old, explain why it is being shown now.

What happened / what is it:
Plain English. Usually 2-4 short sentences.

What's changed / new:
How this differs from prior information, related stories, comparable releases, previous benchmarks, or existing user context. Omit if the comparison would be fake.

Highlights:
- Top thing to know, ordered by importance.
- Stop when remaining details are not worth the user's attention.

Sources:
- [Main source](url) - what it directly supports.
- [Comparison/correction/context source](url) - what it directly supports.
```

## Rules

- Do not fill a quota. One good story is better than several shallow notes.
- Cite the source for every factual claim that is not common context.
- Keep sources out of the main channel when a thread exists.
- Use visuals only when they help the user understand, verify, or compare something.
- Omit sections that would be empty or artificial, except Sources for medium/deep threads.
