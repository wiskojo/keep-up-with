# Writing

Write fact-first updates that save the user time.

## Output Shape

Use this shape for researched updates. Do not force headings when the update is short.

```markdown
Background:
Who is involved? If it is a random startup, add context. If it is a well-known lab or company, keep it short.
When did this happen? If it is old, there should be a reason the user is seeing it now.

What happened:
Plain English. Usually 2-4 short sentences.

What's changed:
How this differs from prior information, related stories, comparable releases, previous benchmarks, or existing user context. Omit if the comparison would be fake.

Highlights:
The top things to know, ordered by importance. Stop when the remaining details are not worth the user's attention.

Sources:
Cite the main source and any sources used for comparison, correction, or context.
```

For a notify-only message, compress this into one natural message.

For a thread, interleave visuals and commentary instead of dumping all sources at the bottom.

## Highlight Selection

Before writing, decide the cutoff.

Include only the most important highlights, ordered by importance.

For each highlight, know:

- what happened
- why this detail is being shown
- what source supports it
- what visual should go with it, if any
- what changed compared with prior information

## Editor Pass

Before sending, check:

- Is the point obvious?
- Is the language natural?
- Is anything padded, vague, or robotic?
- Are claims tied to sources?
- Are visuals useful?
- Is the level of depth appropriate?
- Is the user being told something they would actually want to know?

Revise until it is ready to send.
